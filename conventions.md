# 開発者のクセ・進め方の規約

最終更新: 2026-06-18

開発者個人の作業スタイル。アシスタント（Claude 等）はこれに合わせて動く。

## git: 「コミット」＝ push まで

- **「コミットして」と言われたら commit だけでなく origin への push まで**やる。
- 理由: 開発者は **SVN 育ち**で、commit ＝ サーバ反映（push 相当）の感覚。ローカル commit で止めると「終わっていない」状態になる。
- push せず止めてほしいときだけ、開発者が明示的に「**コミットまでで止めて**」と言う（稀）。
- 複数リポジトリが絡む変更は、関連する全リポジトリを push する。

## ドキュメントの書き方

- **日付は絶対日付**で書く（「先週」ではなく `2026-06-18`）。
- 横展開できる教訓は `knowledge/` に 1 トピック 1 ファイル＋ README 索引に 1 行（詳しくは [README.md](README.md)）。

## 表示文言の変更は全言語そろえる

- 多言語対応するゲームで、**表示文言（UI 文字列）を変えるときは、対応する全言語を併せて直す**。
  片方の言語だけ直して他を取り残さない。変更後はどの言語も意味が一致しているか確認する。

## リリース前チェック（出荷の確実性）

> エンジン・配信先が未確定（TBD）のため、確定したらこの節を具体化する。当面の汎用チェック:

1. **バージョン番号を上げる**（配信のたびに増やす。ストア側のビルド番号と突き合わせられるように）。
2. **広告・課金・解析などのフラグ**を本番用に戻す（テスト用のままリリースしない）。
3. **署名／配信用ビルド**の鍵・設定が今の拠点の値を指しているか（拠点で違う → [environment.md](environment.md)）。
4. 可能ならリリースビルドを**実機で起動確認**してから配信する。
5. 対象プラットフォームの最低動作環境を満たすこと（確定したら具体値をここに書く）。
