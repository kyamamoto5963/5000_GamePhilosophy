# 基本思想（ゲーム開発・beernosusume.net 共通テンプレ）

> 📍 **これが正本（canonical）**。BeerGame 配下の各ゲームはこのファイルを参照する。
> 各プロジェクトの `docs/` 側は、ここへのポインタ＋「そのタイトル固有の適応・判断」だけを書く。

新しいゲームを作るたびに「だいたいどれにも効く」考え方・進め方をまとめた土台メモ。
個々のゲーム仕様ではなく、**プロジェクトの土台として最初に押さえておくもの**。

> ⚠️ **技術スタック（ゲームエンジン）は固定しない**。アプリ開発は Flutter で揃えていたが、
> ゲームは**タイトルごとに要件へ最適なエンジン・構成をそのつど提案・選定する**。
> 姉妹リポジトリ `0000_beernosusumesoft`（Flutter アプリ正本）の技術骨格は**参考程度**で、
> 丸写しにしない。引き継ぐのは「土台の考え方・進め方」であって、特定スタックそのものではない。

---

## 横断思想（プロジェクト間で揃える🟦 / 各自で良い🟨）

| 区分 | 内容 |
|---|---|
| 🟦 共有（揃える） | 開発元規約（beernosusume.net）、リポジトリ命名（リポジトリ名＝フォルダ名）、新規はヒヤリング起点で docs に経緯を残す、命名の一貫性、作業スコープ規律、シンプル基調＋こだわり演出を1つ、2拠点運用の申告ルール、git 作法（コミット＝push） |
| 🟨 個別（各自でOK） | **技術スタックの実装手段**。エンジン・言語・アーキテクチャ・ライブラリはタイトルごとに選んでよい。揃えるのは「何を満たすか（方針）」であって「どう実装するか（手段）」ではない。 |

- 準拠チェックは **方針（🟦）の充足**で判定し、手段（🟨）の一致は問わない。
- 衝突したら：方針（🟦）の差は揃える方向で議論、手段（🟨）の差は各プロジェクトの事情を尊重し**比較材料として残す**。
- **シンプル基調**を共通の美意識とする。要素は極力そぎ落として簡素に保つ。

## 名前を決めながら進める（命名の一貫性）🟦

進める中で、**ドキュメント名・画面（シーン）名・機能（要素）名を「明確に1つに決め」て、以降ブレさせない**。
呼び名が揺れると、人もアシスタント（Claude 等）も「どれの話か」を毎回推測する羽目になり、指示・修正・レビューのコストが上がる。

- 揺れやすい呼称は**どちらか一方に決め、別名は使わない**。
- 決めた名前は **docs に書いて正典化**し、コード（クラス名・コメント）・UI 文言・会話でも同じ語を使う。
- 名前を変えるときは**言い換えを残さず全置換**する（旧名が残ると新たなブレになる）。

## 作業スコープ＝今のプロジェクト＋正本（他プロジェクトは直接いじらない）🟦

あるタイトルで作業しているとき、**変更してよいのは「今いるプロジェクト」と「正本リポジトリ（この GamePhilosophy）」だけ**。
**他の兄弟タイトルのコードや docs を、そのセッションから直接編集しない**。

- **なぜ**: 今ビルド・テストしていないプロジェクトを横から書き換えると、デグレ（意図しない退行）を起こしても気づけない。
- **どうする**: 複数タイトルに効かせたい変更は、正本の [cross-project-changes.md](cross-project-changes.md) に `CPC-NNNN` で**起票するだけ**にとどめる。反映は各プロジェクトの責任で、そのタイトルを次に触るときに適用・テストして記録する。
- **例外**: 開発者が「このタイトルも一緒に直して」と**明示的に指示したときだけ**対象を広げる（その場合も対象を口頭確認する）。

## 開発拠点（家／会社）をセッション開始時に申告する 🟦

開発は**家と会社の2拠点**で行い、**ツールのドライブ/パス・環境変数が拠点ごとに違う**（→ [environment.md](environment.md)）。

- **各セッションの開始時に、開発者がどちらの拠点か（家／会社）を申告する**。アシスタントは申告に応じて拠点依存の設定を切り替える。
- **開始時に申告がなかったら、拠点依存の操作（ビルド・実行・パス参照・署名など）に入る前に必ず問いかける**（「家ですか会社ですか？」）。推測で別拠点のパスを使うと取り違える。
- 拠点で異なる値は**埋め込まず**、その場で確認する（拠点で値が違う）。拠点差はコメントアウトや条件分岐（define）で切り替えられる形にしておく。

## 新規タイトルの進め方（まずヒヤリング → docs に記録）

新しいゲームに着手するときは、**コードより先に「仕様ヒヤリング」から始める**。
思いつきで実装に入らず、背景・要件を会話で引き出し、**あとから経緯を見返せる形で `docs/` に残す**。

> 確立した一般フロー（SDLC／Lean・MVP ディスカバリ／Agile）を個人開発向けに軽量化した版。
> SDLC の骨格に、Lean の「作る前の需要検証」とリリース後の「反復」を足した形。

### フロー

1. **需要をざっと検証**（作る前に）— 「遊びたい人がいるか／自分が遊びたいか」を軽く確かめる。`docs/01_requirements.md` に一段落でも残す。
2. **ヒヤリング** — 何を作るかを対話で固める。最低限:
   - **背景・課題**（なぜ作るのか。これを必ず残す＝後から「なぜこうした」を辿れる）
   - **目的／コアの面白さ**（どんな体験を狙うか）
   - **スコープ**（やること／**やらないこと**を両方）
   - **要件**。各要件に **受け入れ条件（Done の定義）** を一言添える。
   - **想定プレイヤー・プレイの流れ**
   - **前提・方針**: 開発元・対象プラットフォーム・データ保存方針・言語・収益方針など
   - 未確定は「**要確認**」と明記して残す。
3. **docs に起票** — ヒヤリング結果を `docs/01_requirements.md` と `docs/README.md` に書き起こす。
4. **技術スタック（エンジン）提案・選定** — 要件に最適なエンジン・構成をそのつど提案し `docs/05_tech-stack.md` に記録。迷った設計判断は **ADR（決定・理由・却下案を数行）** として残す。
5. **土台を仕込む** — 開発元・命名・環境・プロジェクト構成など。
6. **実装 → リリース準備** — ロードマップのフェーズに沿う。配信前チェックは [conventions.md](conventions.md)。
7. **リリース後: 計測 → 反復** — 反応・不具合を見て backlog に積み、次の一手を決める（尻すぼみ防止）。

### docs/ の標準構成（番号付き・目安）

| ファイル | 内容 |
|---|---|
| `README.md` | docs 索引＋前提・方針（確定事項）＋用語集 |
| `01_requirements.md` | 背景・課題／目的／スコープ／要件／プレイの流れ（ヒヤリングの正本） |
| `02_spec.md` | コアロジック・ゲームルールの仕様 |
| `03_screens.md` | 画面（シーン）構成・UX |
| `04_data-model.md` | データモデル・セーブ/永続化 |
| `05_tech-stack.md` | エンジン選定・パッケージ・プロジェクト構成 |
| `07_roadmap.md` | 開発フェーズ・マイルストーン |
| `09_backlog.md` | 細かい要望・改善メモ（都度追記） |
| `10_philosophy.md` | 本ファイル（正本）へのポインタ＋そのタイトル固有の適応メモ |

- 番号・粒度は目安。**最低でも `README.md` と `01_requirements.md` は最初に作る**（ヒヤリングの記録）。
- 後から要件が変わったら docs を更新し、変更の経緯（なぜ変えたか）も残す。

## 0. 識別子・命名規約

| 項目 | 規約 |
|---|---|
| 開発元 | beernosusume.net |
| リポジトリ／フォルダ | 連番 + 名称（パス都合で改名後も据え置き可。リポジトリ名＝フォルダ名） |
| 識別子（applicationId 等） | エンジン/配信先に応じて決める（アプリ側は `net.beernosusume.<app>` だった。ゲームも踏襲を基本に、配信先要件で判断） |
| プライバシーポリシー | 必要なら beernosusume.net 上に掲示し、ゲーム内＋ストアからリンク |

## シンプル基調＋こだわり演出を1つ 🟦

ゲームもシンプル基調を共通の美意識とする。そのうえで、**そのタイトルの本質を表す「こだわり演出」を必ず1つ**仕込み、
無味乾燥にならない愛着・個性を出す（やり過ぎない・盛らない）。「**1つ入れること自体**＝🟦共有／**演出の中身**＝🟨個別」。
